## Event stream processing

1. Usually we can process each event in isolation but sometimes we need to process a batch of events to get the desired output

2. Processing batches of events are useful in cases like
    - Rate limiting
    - Fraud detection
    - Pattern recognition
    - Recommendations service

3. There are different strategies to consume these events
    - Tumbling window

4. Tumbling window
    - Consumer creates a fixed time interval
    - So all the events coming in that bucket will get processed togather
    - Each event will fall in only 1 window
    - Result is generated based on all the events in a window
    - Result can be anything like an alert, a string value, an SMS, etc (generated by the consumer)
    - Example
        - Checking the number of errors we get in a window while analyzing the log events
        - If the number of errors in a window exceeds the thrashhold value then we can raise an alarm
    - Pros
        - Simple to implement
        - Ideal for periodic data analysis
        - Minimal memory and computation overhead
        - Clear time boundaries
    - Cons
        - We have to wait for the whole window time period to get the result (example we have to wait for the whole window cycle to get the information that there are errors coming on the system)
        - Not good for pattern recognition (as events of 1 pattern can be divided in 2 or more windows)
